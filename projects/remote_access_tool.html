📦 Project Overview

The Secure Remote Access Tool is a Java-based telnet alternative designed to securely execute system commands over a TCP/IP network. It processes terminal commands remotely via ProcessBuilder and features dedicated logic to manage shell built-ins like cd by tracking the per-connection working directory state.

The tool is engineered to mitigate threats such as network eavesdropping, ciphertext tampering, and brute-force attacks by integrating robust cryptographic standards.

🛡️ Security Features & Cryptography

  • Hybrid Key Exchange: Uses RSA-2048 with OAEP padding (SHA-256 + MGF1) to securely exchange AES session keys, protecting against Bleichenbacher-style attacks.
  • Per-Client Session Isolation: Generates a unique AES-256 session key for every client thread, so compromising one session does not impact others.
  • Data Integrity and Confidentiality: Encrypts traffic using AES-256 in GCM mode (no padding). A random 12-byte IV is used per message, and a 128-bit authentication tag detects tampering (e.g., AEADBadTagException).
  • Secure Authentication & Storage: Hashes passwords with PBKDF2-HMAC-SHA256 using 65,536 iterations and a unique 128-bit salt.
  • Timing Attack Prevention: Verifies credentials with constant-time XOR comparisons to avoid leaking match length via timing.
  • Brute-Force & DoS Mitigation: Limits login attempts to 3 per IP using a thread-safe ConcurrentHashMap, enforces a 30-second lockout on failure, and applies a 5-minute idle timeout to reduce resource exhaustion.

🔄 Connection & Execution Workflow

  1. TCP Connection Initialization: The client connects to the server port (default 6600), and the server checks the connecting IP against the lockout list.
  2. Key Exchange Sequence: The server shares its RSA public key (Base64, X.509 format). The client generates a new AES-256 key, encrypts it with the public key, and sends it to the server.
  3. Authentication Phase: The client submits AES-encrypted username and password, and the server validates them against stored PBKDF2 parameters.
  4. Encrypted Execution Loop: After authentication, the client sends AES-encrypted commands. The server logs only the IP and character count (avoiding plaintext logging), executes commands, and returns AES-encrypted terminal output to the client.

🎯 Threat Modeling & Testing Framework

  • MITM Attack Simulation: Includes a Man-in-the-Middle demonstration (MitmDemo.java) that intercepts the key exchange and substitutes the RSA public key, highlighting the need for TLS/SSL authentication or certificate pinning.
  • Security Unit Testing: A dedicated test suite (SecurityUtilsTest.java) validates AES-GCM round trips, Unicode stability, random IV behavior, and authentication tag tamper detection.
